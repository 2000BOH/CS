# BLUECARE 민원관리 웹앱 — 로직 설명 (초보자용)

> 이 문서는 Miro에서 **워크플로우**, **마인드맵** 등을 만들 때 참고할 수 있도록, 앱의 동작을 단계별·구조별로 쉽게 풀어 쓴 설명입니다.

---

## 1. 이 앱이 하는 일 (한 줄 요약)

**호텔/레지던스의 "민원(문의·요청·불편)"을 한곳에서 등록·조회·상태 변경하고, 객실별 이력과 담당자별 업무를 나눠서 보는 웹앱**입니다.

---

## 2. 전체 구조 (3단계로 보기)

```
[사용자]  →  [웹앱 화면]  →  [Supabase DB]
   │              │                    │
   │     로그인·탭 클릭·입력     complaints / rooms 테이블
   │              │                    │
   └──────────────┴────────────────────┘
         데이터는 여기서 일괄 저장·조회
```

- **사용자**: 브라우저에서 로그인 후 각 메뉴(탭)를 눌러 작업합니다.
- **웹앱**: React로 만든 단일 페이지 앱입니다. 탭을 바꿀 뿐, URL이 바뀌는 페이지 이동은 없습니다.
- **Supabase**: 클라우드 DB입니다. **민원(complaints)** 과 **객실 정보(rooms)** 를 저장하고, 앱이 로그인 후 한 번 불러와서 추가·수정·삭제할 때마다 여기와 통신합니다.

---

## 3. 진입점: 로그인 → 보이는 메뉴가 정해짐

1. **로그인 화면**  
   - 아이디: **01 ~ 10** (숫자 2자리)  
   - 비밀번호: 01~09는 `0 + 아이디` (예: 01 → `01`), 08·10은 `1031`

2. **로그인 직후**
   - 앱이 **Supabase**에서 **민원(complaints)** 과 **객실(rooms)** 목록을 한 번 불러옵니다.
   - **누가 로그인했는지(01~10)** 에 따라 **보이는 메뉴(탭)** 가 달라집니다.

| 아이디 | 역할(예시)   | 볼 수 있는 메뉴 |
|--------|--------------|-----------------|
| 01~06, 09 | 일반 운영자 | 입력, 전체조회, 영선, 객실이동, 객실체크, 객실정비, 숙박형태, 객실히스토리, 안내/입력, M01·M02·M03 |
| 07     | 영선 담당    | **영선** 탭만   |
| 08     | 관리사무소   | **객실이동** 탭만 |
| 10     | 키핑팀      | **객실정비** 탭만 |

→ **워크플로우**에서는 "역할별로 접근 가능한 화면이 다르다"는 한 블록으로 두면 됩니다.

---

## 4. 데이터의 두 축: "민원"과 "객실"

앱이 다루는 **핵심 데이터**는 두 가지입니다.

### 4-1. 민원 (complaints)

- **한 건** = 호실 하나에 대한 문의/요청/조치 이력 **한 줄**.
- 예: "101호 에어컨 고장 접수 → 영선팀 배정 → 완료".
- 주요 필드 예:  
  `차수`, `호실`, `구분(CS/영선/입실/퇴실/청소)`, `내용`, `조치사항`, `상태(접수→영선팀→진행중→부서이관/외부업체→완료)`, `등록일시`, `등록자`, `사진` 등.

### 4-2. 객실 정보 (rooms)

- **한 행** = 객실 하나의 **기본 정보**.
- 예: 차수, 호수, 타입, 숙박형태, 임차인, 연락처, 운영종료일 등.
- 민원을 등록할 때 "몇 동 몇 호"를 고르거나, 객실이동/정비 화면에서 이 정보를 보여 주는 데 쓰입니다.

→ **마인드맵**에서는 "민원(이벤트/이력)"과 "객실(마스터)"을 두 개의 큰 가지로 두고, 아래 메뉴들을 하위 가지로 붙이면 됩니다.

---

## 5. 메뉴(탭)별로 하는 일 — 페이지 로직

각 탭은 **같은 민원/객실 데이터**를 **다른 방식으로 보거나, 다른 업무만** 하게 합니다.

| 메뉴         | 하는 일 (초보자용 설명) |
|--------------|--------------------------|
| **입력**     | 새 민원을 등록합니다. 차수·호실·구분·내용·조치사항·상태·연락일·조치일·사진을 넣고 "민원 등록하기"를 누르면 DB에 한 건 추가됩니다. 같은 화면 오른쪽에서 "호실별 민원 검색·조회"로 그 호실 이력을 볼 수 있습니다. |
| **전체조회** | 등록된 민원 전체를 목록/카드로 보고, **상태·구분·날짜**로 필터링합니다. 각 건의 상태를 바꾸거나 내용을 수정할 수 있습니다. |
| **영선**     | 영선 관련 민원만 모아서 보고, 처리일·완료 등 영선팀 업무를 처리합니다. (07번 로그인 시 이 메뉴만 보임) |
| **객실이동** | 퇴실·입주·이사일 등 "객실 이동" 관련 정보를 보고 수정합니다. 퇴실상태(준비→연락→퇴실→계약서→완료), 이사일, 관리사무소 확인 등을 입력합니다. (08번 로그인 시 이 메뉴만 보임) |
| **객실체크** | 객실 점검(퇴실 점검일, 점검자, 체크리스트 등)을 기록합니다. |
| **객실정비** | 청소·정비 예정일, 청소 상태 등 "객실 정비" 업무를 처리합니다. (10번 로그인 시 이 메뉴만 보임) |
| **숙박형태** | 객실별 숙박형태(인스파이어, 장박_개인 등)를 보고 관리합니다. |
| **객실히스토리** | 호실을 고르면 그 호실의 **민원 이력 전체**를 시간순으로 볼 수 있습니다. |
| **안내/입력** | DB 설정 방법 안내, SQL 실행 안내, 그리고 **엑셀/텍스트로 민원 일괄 입력** 기능이 있습니다. (붙여넣기 → DB에 일괄 insert) |
| **M01, M02, M03** | 담당자(예: 동훈·시우·현석)별로 "내가 확인한 업무"를 체크하는 전용 화면입니다. |

→ **워크플로우**에서는 각 메뉴를 블록으로 두고, "입력 → DB 저장", "전체조회 → 필터·상태 변경 → DB 업데이트" 같은 **데이터 흐름**을 화살표로 연결하면 됩니다.

---

## 6. 데이터가 움직이는 흐름 (CRUD)

- **Create(등록)**  
  - **입력** 탭: 폼 작성 후 제출 → `addComplaint` → Supabase `complaints` 테이블에 **insert**.  
  - **안내/입력**: 붙여넣기 → 파싱 → 여러 건을 한꺼번에 **insert**.

- **Read(조회)**  
  - 로그인 직후: `loadComplaints()`, `loadRooms()`로 **complaints**와 **rooms** 전부 select.  
  - 이후에는 이 데이터를 **메모리(React state)** 에 두고, 탭/필터에 따라 **화면만 나눠서** 보여 줍니다. (필요 시 재조회 가능)

- **Update(수정)**  
  - 전체조회·영선·객실이동·객실정비·호실별 조회 등에서 상태/조치사항/날짜 등을 바꾸면 → `updateComplaint` 또는 `updateRoom` → Supabase **update**.

- **Delete**  
  - 현재 앱에서는 "민원 삭제" 버튼이 없어도, DB에서 직접 삭제하거나, 나중에 삭제 기능을 붙일 수 있습니다. (로직상 **삭제 = DB delete** 한 번이면 됨)

→ **마인드맵**에서 "데이터 흐름" 가지 아래에 **Create / Read / Update / Delete**를 넣고, 각각 어떤 화면·함수와 연결되는지 적어 두면 좋습니다.

---

## 7. 화면이 하나인 이유 (SPA)

- 이 앱은 **페이지 URL을 바꾸지 않습니다**.  
- **상태(currentPage)** 로 "지금 어떤 탭이 선택되었는지"만 바꿉니다.  
- 탭을 누르면 → `currentPage`만 변경 → 같은 페이지 위에서 **해당 탭에 해당하는 컴포넌트만** 렌더링됩니다.  
  - 예: `currentPage === '입력'` → `<InputPage />`, `currentPage === '전체조회'` → `<AllComplaintsPage />`.

→ **워크플로우**에서 "탭 선택 = currentPage 변경 = 해당 화면 컴포넌트 표시"를 한 흐름으로 그리면 됩니다.

## 8. Miro에서 쓸 때 추천 구조

### 8-1. 워크플로우

1. **시작**: 로그인(01~10)  
2. **다음**: 역할에 따라 보이는 탭이 제한됨  
3. **각 탭별**:  
   - 입력/수정/필터 등 **사용자 행동**  
   - 그다음 **Supabase로 insert/update**  
4. **공통**: 로그인 시 **complaints + rooms 한 번 로드**

### 8-2. 마인드맵

- **중심**: "BLUECARE 민원관리 웹앱"
- **1단계 가지**:  
  - 로그인·역할  
  - 데이터(민원 / 객실)  
  - 메뉴(입력, 전체조회, 영선, 객실이동, 객실체크, 객실정비, 숙박형태, 객실히스토리, 안내/입력, M01·M02·M03)  
  - 데이터 흐름(CRUD)
- **2단계**: 각 메뉴 아래 "무엇을 보는지 / 무엇을 하는지" 한 줄씩

### 8-3. 간단 플로우 차트 (텍스트)

```
[로그인] → [complaints·rooms 로드] → [탭 선택]
                                           │
     ┌──────────────────────────────────────┼──────────────────────────────────────┐
     ▼                  ▼                   ▼                    ▼                   ▼
 [입력]            [전체조회]            [영선]            [객실이동]           [객실정비] …
  │                   │                   │                   │                   │
  └─ insert           └─ filter/update     └─ update            └─ update           └─ update
       │                   │                   │                   │                   │
       └───────────────────┴───────────────────┴───────────────────┴───────────────────┘
                                            ▼
                                    [Supabase DB]
```

## 9. 요약 체크리스트 (Miro용)

- [ ] 로그인 시 **역할(01~10)** 에 따라 **보이는 메뉴가 다르다**.  
- [ ] 모든 데이터의 **저장소는 Supabase** (complaints, rooms).  
- [ ] **민원** = 호실별 이력 한 건 한 건; **객실** = 호실 마스터 정보.  
- [ ] **입력** = 새 민원 등록(insert); **전체조회·영선·객실이동·객실정비** 등 = 조회·상태 변경(update).  
- [ ] **안내/입력** = 일괄 입력(insert 여러 건) + 설정 안내.  
- [ ] 탭 전환 = **페이지 이동이 아니라** 같은 화면에서 **보이는 컴포넌트만 바꿈**(SPA).

이 체크리스트를 Miro 카드나 스티커로 옮겨 두면, 워크플로우·마인드맵을 그릴 때 빠뜨리지 않고 반영할 수 있습니다.
